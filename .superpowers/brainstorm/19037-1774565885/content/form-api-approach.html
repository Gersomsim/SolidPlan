API de componentes de formulario

¿Cómo se usará cada componente en los templates? Elige el enfoque que te resulte más natural y mantenible.

A

Inputs declarativos

Todo via @Input(). Simple de usar, predecible.

<lib-input
  label="Nombre del proyecto"
  hint="Máximo 100 caracteres"
  placeholder="Ej: Torre Reforma"
  [formControl]="nameCtrl"
  [errors]="{ required: 'Campo obligatorio' }"
/>

<lib-select
  label="Estado"
  hint="Estado actual de la obra"
  [options]="statusOptions"
  [formControl]="statusCtrl"
/>

✓ Pros

  • Muy fácil de usar
  • Autocompletado en el IDE
  • Consistente en todo el proyecto

✗ Contras

  • Menos flexible para casos edge
  • Label siempre es texto plano
B

Content projection (slots)

Label, hint y actions son ng-content con selectores. Máxima flexibilidad.

<lib-input [formControl]="nameCtrl">
  <ng-template libLabel>
    Nombre <span class="text-danger">*</span>
  </ng-template>
  <ng-template libHint>
    Máximo <b>100</b> caracteres
  </ng-template>
</lib-input>

✓ Pros

  • Label puede tener HTML rico
  • Slots completamente custom

✗ Contras

  • Verboso para casos comunes
  • Más código en cada uso
C

Híbrido (recomendado)

Inputs para el 90% de los casos. Content projection para personalización avanzada.

<!-- Caso común: solo inputs -->
<lib-input
  label="Nombre del proyecto"
  hint="Máximo 100 caracteres"
  [formControl]="nameCtrl"
  [errors]="{ required: 'Requerido', maxlength: 'Muy largo' }"
/>

<!-- Caso avanzado: con slots -->
<lib-input [formControl]="nameCtrl" label="Nombre">
  <ng-template libSuffix>
    <lib-icon name="search" />
  </ng-template>
</lib-input>

✓ Pros

  • Simple en casos comunes
  • Flexible cuando se necesita
  • Gradual: crece con las necesidades

✗ Contras

  • Dos patrones a documentar
  • Algo más complejo de implementar
